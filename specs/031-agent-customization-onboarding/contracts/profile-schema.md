# Contract: User Profile Schema

## Overview

This contract defines the structure and validation rules
for the user profile produced by the onboarding agent.
The profile is the primary artifact that connects the
onboarding interview to hero customization.

## File Location

```text
.uf/onboarding/profile.md          # Active profile
.uf/onboarding/history/*.md        # Historical snapshots
```

## YAML Frontmatter Schema

```yaml
version: "1.0.0"          # Required. Semver.
user: "string"             # Required. Defaults to $USER.
status: "draft|complete"   # Required.
created_at: "ISO 8601"     # Required.
modified_at: "ISO 8601"    # Required.
```

### Validation Rules

- `version` MUST be a valid semver string.
- `status` MUST be one of: `draft`, `complete`.
- `created_at` MUST be a valid ISO 8601 timestamp.
- `modified_at` MUST be >= `created_at`.
- `user` MUST be a non-empty string.

## Markdown Body Structure

The body MUST contain exactly four sections in order:

```markdown
## Inspiration
[content]

## Interests
[content]

## Objectives
[content]

## Hero Mapping
[table]
```

### Section Validation

- **Inspiration**: MUST contain at least one sentence of
  narrative content. Empty sections prevent status from
  transitioning to `complete`.
- **Interests**: MUST contain at least one sentence or
  bullet point. Empty sections prevent status from
  transitioning to `complete`.
- **Objectives**: MUST contain at least one numbered or
  bulleted objective. Empty sections prevent status from
  transitioning to `complete`.
- **Hero Mapping**: MUST contain a Markdown table with
  columns: Objective, Hero, Capability, How to Leverage.
  Generated by the onboarding agent after processing
  objectives. May be empty in `draft` status.

## Profile Read Contract

Heroes reading the profile MUST follow these rules:

1. **Check existence first**: If `.uf/onboarding/
   profile.md` does not exist, proceed with default
   behavior. Do NOT error.
2. **Check status**: If `status: draft`, the profile
   is incomplete. Heroes MAY use available dimensions
   but SHOULD note the profile is draft in their output.
3. **Read all dimensions**: Each dimension provides
   different context:
   - Inspiration → tone, values, quality bar
   - Interests → domain, technology, conventions
   - Objectives → priorities, timelines, focus areas
4. **Read hero mapping**: If the hero mapping table
   contains entries for this hero, use them to
   contextualize priorities and focus.

## Profile Write Contract

The onboarding agent writing the profile MUST follow
these rules:

1. **Preserve existing content**: When updating, only
   modify the sections the user explicitly changed.
2. **Create history snapshot**: Before overwriting
   `profile.md`, copy the current version to
   `history/<modified_at>.md`.
3. **Validate before marking complete**: All three
   content sections must have validated content before
   setting `status: complete`.
4. **Update timestamps**: Set `modified_at` to current
   time on every write. Set `created_at` only on first
   creation.
5. **Generate hero mapping**: After processing
   objectives, populate the Hero Mapping table
   automatically. The user may review but the initial
   mapping is agent-generated.

## Backward Compatibility

- When no profile exists: All heroes operate with
  default behavior. Zero regressions. (FR-006, SC-006)
- When profile has `status: draft`: Heroes may use
  available content but must not fail if sections are
  empty.
- When profile `version` is newer than the hero expects:
  Heroes SHOULD read what they can and ignore unknown
  fields. No hard version gate.

## Gitignore Considerations

The `.uf/` gitignore block (managed by `uf init`)
currently excludes runtime data:

```gitignore
.uf/**/*.db
.uf/**/*.db-shm
.uf/**/*.db-wal
.uf/**/*.lock
.uf/**/*.log
```

Profile files (`.uf/onboarding/profile.md` and
`.uf/onboarding/history/*.md`) are NOT excluded by
these patterns and SHOULD be committed to version
control, since they represent intentional user
configuration rather than runtime state.
