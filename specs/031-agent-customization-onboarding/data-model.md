# Data Model: Agent Customization Onboarding

## 1. User Profile

The core entity produced by the onboarding agent. Stored
as a Markdown file with YAML frontmatter, following the
same pattern as Muti-Mind's backlog items (per Spec 004).

**Storage**: `.uf/onboarding/profile.md`

### Fields (YAML Frontmatter)

- `version` (string): Profile schema version (e.g.,
  `1.0.0`). Follows semver.
- `user` (string): User identifier. Defaults to the
  system username (`$USER`). Used for future multi-user
  support.
- `status` (enum): `draft`, `complete`.
  - `draft`: One or more dimensions are incomplete or
    the user has not finalized the profile.
  - `complete`: All three dimensions populated and
    confirmed by the user.
- `created_at` (timestamp): ISO 8601 creation time.
- `modified_at` (timestamp): ISO 8601 last modification
  time.

### Body (Markdown)

The body contains three required sections and one
optional section:

#### Section: Inspiration

What motivates the user, what "great" looks like for
their project, and the aspirational vision they want
the AI swarm to work toward.

```markdown
## Inspiration

[Freeform narrative. May include bullet lists,
references to projects, products, or principles
the user admires. The onboarding agent elicits this
through questions like "What does success look like?"
and "What projects or teams inspire you?"]
```

#### Section: Interests

The user's domain focus, technology preferences, and
areas they want to invest time in. This informs which
conventions, patterns, and priorities the heroes
should emphasize.

```markdown
## Interests

[Freeform narrative. May include technology stacks,
problem domains, architectural preferences, and
learning areas. The onboarding agent elicits this
through questions like "What technologies excite you?"
and "What domain are you working in?"]
```

#### Section: Objectives

Concrete goals, timelines, and success metrics. This
is the most structured section because it maps directly
to hero capabilities.

```markdown
## Objectives

[Structured list of goals. Each objective should be
specific enough to map to a hero capability. The
onboarding agent elicits this through questions like
"What are the top 3 things you want to accomplish?"
and "How will you know you've succeeded?"]
```

#### Section: Hero Capability Mapping (auto-generated)

Generated by the onboarding agent after processing the
user's objectives. Maps each stated objective to the
hero(es) best suited to support it.

```markdown
## Hero Mapping

| Objective | Hero | Capability | How to Leverage |
|-----------|------|------------|-----------------|
| [User's objective] | [Hero name] | [Specific capability] | [Actionable guidance] |
```

## 2. Profile History Entry

A timestamped snapshot of a previous profile version,
created whenever the user updates their profile.

**Storage**: `.uf/onboarding/history/YYYY-MM-DDTHH-MM-SS.md`

### Fields

Identical to the User Profile entity. The snapshot is a
full copy of the profile at the time of update. The
filename encodes the timestamp.

### History Behavior

- When the onboarding agent updates a profile, it copies
  the current `profile.md` to
  `history/<modified_at timestamp>.md` before writing
  the updated version.
- History files are immutable once created.
- The `.uf/` gitignore rules exclude runtime data
  (databases, caches, logs) but profile files are
  user-generated content and SHOULD be committed to
  version control.

## 3. Hero Capability Reference

A static reference structure embedded within the
onboarding agent file. Not persisted separately.

### Fields

- `hero` (string): Hero name (e.g., "Muti-Mind").
- `agent_file` (string): Path to agent file (e.g.,
  `.opencode/agents/muti-mind-po.md`).
- `capabilities` (array of strings): Core capabilities
  the hero provides.
- `routing_examples` (array of strings): Example user
  objectives that should route to this hero.

### Reference Data (embedded in agent file)

| Hero | Agent File | Capabilities | Example Objectives |
|------|-----------|--------------|-------------------|
| Muti-Mind | `muti-mind-po.md` | Backlog management, prioritization, acceptance authority | "Prioritize work", "Manage backlog", "Define what to build" |
| Cobalt-Crush | `cobalt-crush-dev.md` | Feature implementation, coding conventions, spec-driven development | "Write clean code", "Implement features", "Follow conventions" |
| Gaze | (binary) | Quality analysis, test coverage, CRAP scores, static analysis | "Improve test coverage", "Check code quality", "Find risky code" |
| The Divisor | `divisor-*.md` (5 personas) | Code review (Guard: drift, Architect: structure, Adversary: security, SRE: operations, Testing: tests) | "Review code", "Security audit", "Check architecture" |
| Mx F | `mx-f-coach.md` | Metrics, coaching, sprint management, retrospectives, impediment tracking | "Track velocity", "Run retrospective", "Manage sprints" |

## 4. State Transitions

### Profile Status

```text
(no profile) ──[/onboard]──> draft
   draft ──[all dimensions complete + user confirms]──> complete
   complete ──[user re-invokes /onboard to update]──> complete
   draft ──[user abandons]──> draft (persisted as-is)
   draft ──[user re-invokes /onboard]──> resumes from last section
```

### Dimension Completion

Each of the three dimensions (inspiration, interests,
objectives) tracks completion independently:

- **Empty**: Section heading exists but no content below
  it.
- **Populated**: Content exists below the section
  heading.
- **Validated**: The onboarding agent has confirmed the
  content is actionable (not vague or generic).

A profile's status transitions to `complete` only when
all three dimensions are populated AND validated.
